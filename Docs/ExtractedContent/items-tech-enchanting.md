## Major design buckets

| Bucket | Examples | Target system |
|---|---|---|
| Currency/resources | Money, Planet_Money, resource deposits | wallet/resource components + persistence |
| Basic items | Senzu, weights, wearable items | item prototypes + item instances |
| Weapons | Sword, Gun, NGun, ammo | equipment slots + attack/projectile integration |
| Armor/equipment | Armor, wearable equipment | equipment stat modifiers, durability |
| Buff/transform items | StoneMask, BuffItem, TransEmblem | high-risk custom item effects |
| Tech machines | AutoDrill, PunchingBag, Simulator, Regenerator, Safe | buildable/machine systems |
| Communications/security | Communicator, Scouter, Doorbell, Speaker, Camera, Drone, SecurityDisplay | device channels, ownership, permissions |
| Doors/forcefields | ReinforcedDoor, remotes, force field emitters | access control, map entities |
| Vehicles/space travel | ShipConsole, PodConsole, SpacePod/Ship accessories | later vehicle/map travel system |
| Enchantment machines | Cauldron, InfusionCrystal, Arcanic devices | enchanting/crafting recipes |
| Potions/consumables | Healing/Energy/Mana/HiPotion/Ether/Elixer/BuffPotions | consumable effect handlers |
| Magical travel/storage | Portal, LinkingBook, PocketDimensionGenerator | teleport/storage systems with strong validation |

## Rewrite guidance

- Items should have prototypes; only unique mutable data should be item instance state.
- Equipping should add stat modifiers through the same stat system as transformations.
- Crafting/enchanting should be server-validated recipes, not client-driven `input` flows.
- High-risk systems such as portals, pocket dimensions, drones, turrets, ships, and forcefields should wait until core persistence and permissions are stable.

## Test cases

- Equip/unequip applies/removes modifiers once.
- Item instance persists only explicit fields, not EntityUid references.
- Craft/enchant rejects missing resources/ownership/range.
- Turret/drone/forcefield cannot be controlled without authorization.
